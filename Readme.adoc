[#_project_structure]
== 📦 Project Structure

This is a multi-module project consisting of::

* *link:./imperative-throughput[imperative-throughput]*: Traditional Spring MVC implementation using blocking operations
* *link:./reactive-throughput[reactive-throughput]*: Spring WebFlux implementation using non-blocking reactive streams

  -Dmaven.test.skip=true \
  -Dmaven.build.cache.enabled=false \

[#_comparison]
== ⚖️ Paradigm Comparison

[cols="2,3,3", options="header", width=100%, frame=all, grid=rows, title="Imperative vs Reactive — Design & Behaviour"]
|===
| Dimension | Imperative + Virtual Threads | Reactive (WebFlux + Reactor)

| *Concurrency model*
| One virtual thread per request. The JVM multiplexes millions of virtual threads onto a small carrier-thread pool. Blocking a virtual thread parks it without consuming a carrier thread.
| One or few Netty event-loop threads handle all I/O via non-blocking callbacks and operator pipelines. No thread is ever blocked.

| *Blocking I/O*
| Safe — blocking unmounts the virtual thread from its carrier and remounts it when I/O completes. `RestClient`, `JDBC`, `Files` work unchanged.
| Forbidden on event-loop threads. All I/O must go through non-blocking drivers (`WebClient`, R2DBC, reactive file I/O).

| *Programming model*
| Standard sequential Java — `try/catch`, `for` loops, debuggable stack traces. Existing blocking libraries require no changes.
| Functional pipeline — `Mono`, `Flux`, operators. Requires learning Project Reactor; traces need the Reactor debug agent to be readable.

| *Parallel fan-out*
| `StructuredTaskScope` (JEP 505, Java 25 preview) — fork N tasks, join with structured lifecycle and automatic failure propagation.
| `Flux.merge()` — merge N concurrent `Mono` pipelines; no threads are created.

| *Timeout + fallback*
| `CompletableFuture.orTimeout(500, MILLISECONDS)` + `.handle()` to substitute a static fallback on `TimeoutException`.
| `Mono.timeout(Duration.ofMillis(500))` + `.onErrorReturn(fallback)` — inline operator, zero extra threads.

| *Backpressure / SSE*
| Not natively supported. A thread-per-connection model cannot propagate consumer-side backpressure to the producer without blocking the producer thread.
| `Flux.generate` + `.delayElements()` — demand-driven emission. The subscriber controls pace. Used for `GET /stream` (reactive only).

| *Error handling*
| `@ControllerAdvice` + `@ExceptionHandler(RuntimeException.class)`, `HttpServletRequest` for path extraction.
| `AbstractErrorWebExceptionHandler` (`@Order(-2)`) composing a `ServerResponse`.

| *Error response contract*
| `{"status":500,"error":"Internal Server Error","path":"..."}`
| `{"status":500,"error":"Internal Server Error","path":"..."}` (unified)

| *Scheduler / thread pool*
| `Executors.newVirtualThreadPerTaskExecutor()` (per-request scope); `Thread.ofVirtual().factory()` for the smoke delay scheduler.
| Reactor `boundedElastic` for CPU work; Netty event loop for I/O; shared `virtualThreadScheduler` bean for reactive smoke delay.

| *Observability*
| Custom `jvm.threads.virtual.active` Micrometer gauge (sampled from `ThreadMXBean`) — shows virtual-thread count scaling under load.
| Standard Micrometer timers per endpoint tagged `module=reactive`. Event-loop thread count stays constant regardless of load.

| *Test approach*
| `@WebMvcTest` + `MockMvc` with `asyncStarted` / `asyncDispatch` for `CompletableFuture` responses.
| `@SpringBootTest(RANDOM_PORT)` + `WebTestClient` + `StepVerifier` for reactive pipelines.

| *HTTP server*
| Apache Tomcat 11
| Reactor Netty

| *Base path*
| `/imperative-throughput` (servlet `context-path`)
| `/reactive-throughput` (`spring.webflux.base-path`)

| *Default port*
| `8888`
| `9999`
|===

[cols="2,1,1", options="header", width=75%, frame=all, grid=rows, title="Endpoint Coverage"]
|===
| Endpoint | Imperative | Reactive

| `GET /smokes` — 300 ms baseline latency, returns thread identity
| ✅ | ✅

| `GET /cpu` — Fibonacci(50) CPU-bound workload
| ✅ | ✅

| ✅ | ✅

| ✅ | ✅

| `GET /stream` — SSE, 10 events × 200 ms, backpressure-aware
| ❌ | ✅
|===

=== What the Numbers Actually Say

[cols="2,5", options="header", width=100%, frame=all, grid=rows]
|===
| Finding | Evidence

| *Parity below ~2 500 concurrent threads*
| Both deliver ~272 TPS, ~302 ms median, zero errors.
  Virtual threads handle moderate I/O-wait load as efficiently as a non-blocking event loop.

| *Carrier-thread saturation is the virtual-thread ceiling*
| At 5 000t × 25 loops the imperative mean jumps 2.4× (302→731 ms) while reactive stays flat (326 ms).
  Median remains 302 ms in both — the distribution becomes bimodal as carrier threads queue.
  This is the pinning/saturation boundary, not a general virtual-thread weakness.

| *Reactive is the only zero-error option under maximum sustained load*
| At 2 M samples (5 000t × 200 loops) imperative produces 13 335 failures (1.33%) and a 31 s worst-case
  spike; reactive holds 0 errors and caps at 2.4 s. Aggregate TPS is identical (~540 each), proving
  the bottleneck is reliability, not capacity.

| *Virtual threads don't prevent tail-latency spikes once carrier threads pin*
| Imperative p99: 314 ms (light) → 625 ms (5 000t burst) → 472 ms (sustained).
  Reactive p99 stays within 340–472 ms across all five runs.

| *Native build = startup + memory, not runtime throughput*
| GraalVM native cuts startup from >3 s to <1 s and reduces heap usage — valuable for
  container-dense or serverless deployments. Steady-state TPS and latency are set by the
  concurrency model, not the execution engine.

| *Choose by concurrency contract, not by hype*
| Use *imperative + virtual threads* when peak concurrency stays below ~2 500: simpler code,
  familiar debugging, identical measured performance.
  Use *reactive* when you must guarantee zero errors and bounded tail latency beyond that threshold,
  or when you need native SSE/backpressure (`/stream` endpoint).
|===
